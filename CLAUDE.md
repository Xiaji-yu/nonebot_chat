# 项目上下文获取机制

## 缓存架构

本项目采用**静态规范 + 动态缓存**分离架构：
- **CLAUDE.md（静态层）**：定义通用的开发规范和行为准则
- **运行时缓存（动态层）**：通过项目分析自动生成，存储项目特定的技术栈、测试命令等上下文信息

**设计优势：**
- 规范文件可跨项目复用，无需为每个项目修改
- 上下文自动更新，避免手动维护导致的过时问题
- 单职责分离：CLAUDE.md 负责"应该怎么做"，缓存负责"这个项目是什么"

## 自动刷新策略

### 触发条件

缓存会在以下文件变更时自动标记为过期（下次对话时重新生成）：

**Python 项目：**
- `pyproject.toml`, `setup.py`, `setup.cfg`, `requirements*.txt`

**Node.js 项目：**
- `package.json`, `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`

**Go 项目：**
- `go.mod`, `go.sum`

**测试配置：**
- `pytest.ini`, `.pytest.ini`, `tox.ini`
- `jest.config.js`, `vitest.config.ts`

**构建与部署：**
- `Dockerfile`, `docker-compose.yml`, `Makefile`

**Claude 专用：**
- `.claude/settings.json`, `.claude/skills/` 目录

### 失效机制

- **TTL 过期：** 缓存生成超过 7 天自动标记为过期
- **文件变更：** 检测到上述文件的修改时间晚于缓存生成时间
- **手动刷新：** 用户可随时运行 `/refresh-context` 强制更新

## 降级策略

当缓存不存在、损坏或分析失败时：

1. **尝试使用上次有效缓存**（如果存在且未过期）
2. **回退到保守默认值：**
   - 测试命令：基于文件扩展名推断（检测到 `pytest.ini` 则用 `pytest`）
   - 技术栈：基于文件扩展名统计（`.py` → Python，`.ts` → TypeScript）
3. **用户提示：** 在回复中显示 `⚠️ 项目上下文缓存过期或损坏，正在使用降级模式。运行 /refresh-context 更新缓存`

# 开发规范

本文件约束 Claude 在此项目中的所有行为。**所有规范强制执行，不可被用户的单次指令覆盖。**

## 沟通原则

- 中文沟通，回复简洁直接，不过度解释
- 发现更优路径时主动建议，不只按指令做
- 执行前先给出分析报告（典型问法是**"有必要吗？"**），包含：
  - 当前现状（问题是否真实存在）
  - 修复/新增的收益（量化：影响哪些场景）
  - 不做的风险
  - **明确推荐**（首选方案后标注"（推荐）"）

---

## 一、动手前的原则

### 1.1 先理解再修改
- 修改任何文件前，必须先 Read 该文件的完整内容
- 修改超过 3 个文件的任务，先用 EnterPlanMode 制定方案，获得批准后再执行
- 不确定影响范围时，先用 Grep/Glob 搜索相关调用方

### 1.2 最小化改动
- 只修改任务涉及的代码，不"顺手"重构无关部分
- 不引入任务未要求的新依赖
- 保持现有的代码风格和命名约定

### 1.3 设计决策前的深度问答

在开始实现新功能或修改重要模块之前，必须主动追问决策节点。
不要假设用户已经想清楚了所有细节——大多数返工都源于"先写代码再理解"。

**结构类问题（决定代码放哪、怎么组织）：**
- 这个改动涉及哪些现有模块？
- 新代码应该放在哪个目录/文件？
- 与最近的相似实现相比，这个有什么不同？
- 有没有已有的模式/约定我应该遵循？

**行为类问题（决定代码做什么）：**
- 成功时的输出/行为是什么？失败时呢？
- 边界条件有哪些（空输入、极端值、并发）？
- 调用方是谁？会被谁依赖？

**风险类问题（决定要不要做、怎么做安全）：**
- 如果这个改动出问题，影响范围有多大？
- 有没有设计疑问可以先跑个原型验证？

**规则：**
- 一次问一个问题，等用户回答后再问下一个
- 如果某个问题可以通过探索代码库回答（如"有没有已有模式"），先探索再问
- 用户说"你定就行"时，给出推荐方案并说明理由，然后继续

## 二、Bug 修复规范

### 2.1 修复前必须理解根因
- 说清楚：**为什么会出错**（不只是"哪里出错"）
- 考虑所有合法的边界状态，不只是最常见路径

### 2.2 修复后必须附带回归测试
- 每个 Bug 修复都要新增一个测试，专门验证这个场景不会再次出现
- 测试命名格式：`test_<被修复的行为描述>`

### 2.3 平台差异必须显式处理
- 涉及文件路径、换行符、编码的代码，必须显式处理 Windows/Linux 差异
- 不依赖"在我机器上能跑"的假设

---

## 三、代码质量规范

### 3.1 类型注解
- 所有函数参数和返回值必须有类型注解（包括返回 `None` 的函数）
- 使用语言推荐的现代写法（Python: `X | None`；TypeScript: `X | null`）

### 3.2 文档注释
- 所有**公共**函数、方法、类必须有文档注释
- 使用该语言的标准风格（Python: Google 风格 Docstring；JS/TS: JSDoc）
- **不写显而易见的内容**

### 3.3 常量管理
- 代码中**禁止出现魔法数字和魔法字符串**
- 命名全大写加下划线，定义旁必须有注释说明含义和来源
- 常量组织策略：
  - 单模块项目：集中定义在 `constants.py` / `constants.ts`
  - 多模块项目：每个模块内部定义私有常量；跨模块共享的常量集中放 `common/constants.py`（或 `shared/constants.ts`），禁止循环引用

### 3.4 消除重复代码
- 同一段逻辑出现 **2 次以上**就提取为函数或方法
- 提取的函数命名要体现意图，而不是实现细节

### 3.5 导入/引用顺序
- 遵循语言或工具的标准约定（Python: ruff/isort；JS/TS: ESLint import order）
- 禁止在函数内部写导入，除非解决循环依赖

### 3.6 异常处理
- **禁止**裸捕获或无日志的静默吞异常
- 捕获具体异常类型优先于通用 Exception
- 意料外的异常必须记录完整堆栈（`logger.error(..., exc_info=True)`）
- 日志级别对应关系：

  | 情况                     | 级别                   |
  | ------------------------ | ---------------------- |
  | 已知业务错误，有降级处理 | `warning`              |
  | 意料外的异常             | `error(exc_info=True)` |
  | 影响整个系统的致命错误   | `critical`             |

## 四、性能规范

### 4.1 热路径审查
- 每条消息/请求都会调用的函数是**热路径**，必须评估时间复杂度
- 避免在热路径上全量遍历（O(n)），优先使用 O(1) 数据结构

### 4.2 不变数据只读一次
- 环境变量、配置文件、白名单：首次加载后缓存，不重复读取
- 缓存变量命名加 `_cache` 后缀，并在注释中说明失效策略

### 4.3 算法选择要有依据
- 选择数据结构时用注释说明理由（为什么用这个，不用那个）

## 五、测试规范

### 5.1 新功能必须附带测试
- 新增功能 → 同时新增测试，不接受"后面再补"
- Bug 修复 → 必须附带回归测试

### 5.2 测试隔离
- 测试不依赖网络、数据库、外部服务
- 用 Mock/Stub 模拟异步调用和外部依赖
- 用环境隔离机制隔离环境变量，不修改全局状态

### 5.3 测试覆盖范围
每个模块的测试必须覆盖：
- ✅ 正常路径（happy path）
- ✅ 边界条件（空输入、最大值、特殊字符）
- ✅ 错误路径（API 失败、网络超时、非法参数）
- ✅ 状态机转换（如果有状态的话）
- ✅ 并发/竞争条件（如果涉及共享状态、异步操作或锁）

### 5.4 测试命名规范
- 格式：`test_<被测行为>_<条件>`
- ✅ `test_api_call_raises_on_nonzero_retcode`（描述行为）
- ❌ `test_circuit_breaker_method`（描述实现）

### 5.5 运行测试
- 修改后必须确保全部通过，才能进行下一步
- 具体命令见**项目上下文**

## 六、可观测性规范

### 6.1 关键路径必须有指标
- 消息收到/发出/失败的数量
- 错误发生的次数和时间
- 操作的延迟（用百分位 p50/p95，不用平均值）

### 6.2 下游依赖必须有保护
- 必须有**超时**设置
- 连续失败超过阈值时必须有**熔断**（快速失败，不继续发请求）
- 熔断恢复要有探测机制（HALF_OPEN 状态）

### 6.3 告警要有冷却时间
同一条告警规则触发后，冷却期内不重复触发，避免告警风暴。

### 6.4 状态可查询
运行时状态必须有查询入口（如 `/status` 命令或健康检查端点），不能只有"翻日志文件"一种方式。

## 七、提交规范

### 7.1 Conventional Commits 格式
```
<type>(<scope>): <subject>

[可选 body：说明为什么，不是说明做了什么]

Co-Authored-By: Claude <noreply@anthropic.com>
```

**scope 取值规则**：
- 单模块项目：使用文件名或功能模块名（如 `auth`、`parser`、`db`）
- 多模块项目：使用顶层目录名（如 `api`、`worker`、`shared`）
- 影响全局的改动：scope 留空，即 `feat: subject`
- 禁止使用过于宽泛的 scope（如 `code`、`misc`、`update`）

| type       | 用途             |
| ---------- | ---------------- |
| `feat`     | 新功能           |
| `fix`      | Bug 修复         |
| `docs`     | 仅文档           |
| `test`     | 测试相关         |
| `refactor` | 重构（不改功能） |
| `perf`     | 性能优化         |
| `chore`    | 构建、依赖、配置 |

### 7.2 提交工作流
- **每次完成一个阶段后立即 commit**，保持提交粒度细、语义清晰
- **绝不主动 push**，push 由用户在确认后手动执行
- 大型改动按**语义**拆分，每个 commit 只做一件事

### 7.3 提交前检查
- 运行 pre-commit 钩子（`pre-commit run --all-files`），必须全部通过才能提交

## 十一、工作习惯偏好

### 11.1 功能取向：轻量优先，零外部依赖
- 可观测性：纯内存 + 标准库，不引入 Prometheus/Grafana
- 告警：写日志 + `/status` 命令，不引入消息队列
- CI：本地 pre-commit 钩子，**不用 GitHub Actions**
- 新增依赖前必须征得用户同意；能用标准库解决的，不引入三方包

### 11.2 测试哲学：按价值选测试目标，不追求覆盖率数字

| 值得测试                             | 不值得测试               |
| ------------------------------------ | ------------------------ |
| 纯逻辑（无 I/O）的状态机、分支、边界 | I/O 协调层               |
| 有历史 bug 记录的路径                | mock 层数超过 3 层的场景 |
| 被多处调用的关键工具函数             | 只是 wrap 一个库调用     |

### 11.3 代码审查：双轴框架 + 完成自检清单

审查时沿两个独立维度检查，不要混在一起：

**Axis 1 — Standards：代码是否遵循了仓库的编码规范？**
- 类型注解、命名约定、导入顺序、异常处理是否符合第三~七章规范
- 常量是否全大写、是否有魔法数字
- 提交信息是否符合 Conventional Commits

**Axis 2 — Spec：代码是否忠实实现了需求？**
- 行为是否与 Issue/PRD 描述一致
- 是否有多做（gold plating）或少做（遗漏边界）
- 接口签名是否与约定匹配

两个轴都通过才算合格。Axis 2 不过而 Axis 1 过 → 代码写得漂亮但做错了事；Axis 1 不过而 Axis 2 过 → 功能对了但技术债累积。

**完成后主动检查以下问题：**
- **死代码**：赋值后从未使用的变量
- **缺失类型注解**：参数写了默认值但没有类型
- **静默丢弃**：`else: pass` 或 `logger.debug` 处理了本应提取的信息
- **幂等性**：重复执行脚本时是否安全
- **接口最小化**：对外暴露的函数/属性是否超过调用方实际需要的范围
- **并发安全**：涉及共享状态时，是否有竞争条件；锁的粒度是否合理（不过宽、不过细）
- **函数签名稳定性**：新增参数是否有默认值；破坏性改动是否已知会影响调用方

### 11.4 改动范围控制：一次只做一件事
- 每个 commit 对应一个明确的语义单元
- 不"顺手"重构与当前任务无关的代码
- 大改动（>3 个文件）先进 EnterPlanMode，获得批准再执行
- 用户随时可以说"先停在这里，帮我 commit"，Claude 应立即响应

### 11.5 依赖管理
- 引入任何新三方依赖前必须征得用户同意，并评估：
  - 维护状态（最近是否活跃、有无 abandoned 标记）
  - 许可证（MIT/Apache-2.0 优先；GPL 系需确认是否影响分发）
  - 包大小与传递依赖数量
- 能用标准库解决的，不引入三方包
- 版本锁定策略：
  - Python：`requirements.txt` / `poetry.lock` pin 到 patch 版本（如 `1.2.3`，不用 `>=`）
  - Node.js：提交 `package-lock.json` / `yarn.lock`，不用 `*` 或 `latest`
- 定期（每个 milestone 结束）运行安全扫描（`pip audit` / `npm audit`），有高危漏洞必须处理后才能发布

## 十二、分支策略

### 12.1 分支模型
- 主分支：`main`（受保护，只能通过 PR 合并）
- 功能分支：`feature/<short-name>`（从 main 切出）
- 热修复：`hotfix/<issue-number>`（从 main 切出）
- 实验分支：`experiment/<name>`（从 main 切出，验证后删除或合并）

### 12.2 命名约定
- 功能分支：`feature/add-user-auth`、`feature/refactor-db-layer`
- 修复分支：`fix/42-login-timeout`、`hotfix/89-crash-on-startup`
- 实验分支：`experiment/new-cache-strategy`

### 12.3 PR 规范
- 标题格式：`<type>(<scope>): <subject>`（同提交规范 7.1）
- PR 描述必须包含：
  - 变更摘要（1-3 句话，说明改了什么、为什么改）
  - 关联 Issue（`Closes #123` 或 `Relates #123`）
  - 测试说明（改了哪些功能，如何验证）
  - 截图（UI 变更必须附截图）
- PR 合并前必须通过：
  - [ ] pre-commit 钩子全绿
  - [ ] 全量测试通过
  - [ ] 至少一轮 review

### 12.4 合并策略
- 功能分支 → `squash and merge`（保持 main 线性历史）
- 热修复 → `merge commit`（保留完整时间线，便于追溯）
- 实验分支 → squash merge 或关闭（不保留实验历史）

### 12.5 生命周期
- 功能分支合并后立即删除
- 超过 7 天未合并的分支，检查是否还相关，不相关则关闭
- 禁止在 main 上直接 push（只能通过 PR）

# 附录

## 附录 A、领域语言（CONTEXT.md 模式）

> **来源：Matt Pocock 最推崇的技法**。通过建立项目专属术语表，解决 AI 每次 session 都要重新理解项目 jargon 的问题。

### 什么是 CONTEXT.md

每个项目根目录下的 `CONTEXT.md`，记录该项目的**领域术语**和**关键决策**。

### 为什么需要它

- 没有它：AI 每次新 session 都要从零理解项目术语
- 有了它：AI 直接用统一语言沟通，减少 30%+ token 消耗
- 副作用：变量名、函数名、文件名自动保持一致

### 内容格式

```markdown
# CONTEXT.md

## 核心概念
- **模块名** — 一句话定义，说明职责
- **另一个概念** — 一句话定义

## 业务术语
- **业务词** — 在项目中的具体含义（不是字典定义）

## 架构决策
- **ADR-0001** — 简短说明（链接到 docs/adr/ 完整版）
```

### 维护规则
- 新术语出现时**立即添加**，不要等"整理好了再写"
- 修改 CONTEXT.md 等同于修改代码——需要 review
- 如果 AI 在对话中创造了新术语，检查是否需要纳入 CONTEXT.md
- 如果某个概念需要解释实现细节才能说清，那些细节属于 ADR 或代码注释，不属于 CONTEXT.md

### ADR 创建三条件（必须同时满足）

写 ADR 之前，检查是否三个条件都满足：

1. **难以回滚** — 改变主意的代价显著（不是"改一行代码就能换"的事）
2. **不看会困惑** — 一个 future reader 看到这段代码会问"为什么这样做？"
3. **真实权衡** — 真的比较过 A 和 B，选了其中一个，有具体原因

**三个都满足 → 写 ADR。**
**缺任何一个 → 不写，decision 记录在 commit message 就够了。**

示例判断：
- "选了 PostgreSQL 而不是 MongoDB" → 三个都满足 → 写 ADR
- "用了 dict 而不是 list 做 lookup" → 条件 1 不满足（一行代码就能换）→ 不写
- "把函数名从 getData 改成 fetchUserProfile" → 条件 2 不满足（名字本身解释了一切）→ 不写

### 纯净性边界（必须遵守）
- **CONTEXT.md 是术语表，不是规格说明书。** 不写实现细节、不写文件路径、不写代码片段
- **不是草稿纸。** 不在里面记录临时的实现决策，这些属于 ADR 或 commit message
- **不是功能清单。** 不在里面积累"要做的事情"，这些属于 Issue tracker
- 违反边界 = 文件逐渐变成垃圾箱，失去作为领域语言加速器的作用

## 附录 B、结构化方法论

> **来源：Matt Pocock 的 disciplined engineering skills**。将原则转化为可执行的协议，防止"知道但做不到"。

### B.1 调试协议（Diagnose Loop）

遇到 Bug 时，**按顺序执行，不跳步**：

```
Phase 1 — 建反馈循环（最重要！）
  ↳ 写一个最小复现脚本/测试，30 秒内能触发 bug
  ↳ 没有循环 = 没有调试，只是猜

Phase 2 — 复现
  ↳ 运行循环，确认 bug 可稳定复现
  ↳ 确认复现的是用户描述的 bug，不是旁边的另一个 bug

Phase 3 — 假设（列出 3-5 个）
  ↳ 每个假设必须可证伪："如果 X 是原因，那么改 Y 会让 bug 消失"
  ↳ 单假设 = 锚定效应，会错过真正原因

Phase 4 — 仪器
  ↳ 一次只改一个变量
  ↳ 用 debugger/REPL > 日志（一个断点胜过十条日志）
  ↳ 日志必须加唯一前缀 [DEBUG-xxx]，事后 grep 清理

Phase 5 — 修复 + 回归测试
  ↳ 先写测试（会失败）→ 再修复 → 看测试通过
  ↳ 回归测试必须能锁住这个 bug pattern

Phase 6 — 清理
  ↳ 移除所有 [DEBUG-xxx] 日志
  ↳ 删除临时原型文件
  ↳ 确认原始场景已修复
  ↳ commit message 中注明"哪个假设被证实了"（帮助下次调试者）
```

**关键原则：** 没有 Phase 1 的反馈循环，不要进入 Phase 2。

### B.2 TDD 反模式：水平切片

```
❌ WRONG — 水平切片
  RED:  test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

✅ RIGHT — 垂直切片（Tracer Bullet）
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
```

**为什么水平切片是反模式：**
- 批量写的测试测的是「想象的行为」，不是「实际行为」
- 测试对真实变化不敏感（行为变了但测试照样过）
- 在写实现之前就承诺了测试结构，缺乏信息时做决定

**Tracer Bullet 的核心：** 每个 RED-GREEN 循环都是一次完整的学习——你刚写了代码，所以你知道什么行为真正重要。

### B.3 原型协议（Prototype）

> **原则：原型是回答一个问题的废弃代码。问题决定形状。**

#### 两条分支

| 用户的隐含问题 | 分支 | 产出 |
|--------------|------|------|
| "这个逻辑/状态模型感觉对吗？" | 逻辑原型 | 终端交互式 app，推状态机通过难推理的场景 |
| "这个界面应该长什么样？" | UI 原型 | 多个 UI 变体，URL 参数切换 |

选错分支 = 浪费整个原型。如果问题模糊且用户不在线，默认选与周围代码匹配的分支（后端模块 → 逻辑；页面/组件 → UI），并在开头注明假设。

#### 六条铁律

1. **第一天就标记为 throwaway。** 原型代码放在真实位置旁边（上下文明显），但文件名要让读者一眼看出是原型
2. **一条命令可运行。** 用项目已有的 task runner（`pnpm <name>`、`python <path>` 等），用户不用想就能启动
3. **默认无持久化。** 状态活在内存里。持久化是原型要**验证**的事，不是它要**依赖**的事
4. **跳过打磨。** 无测试、无超出运行所需的错误处理、无抽象。目的是快速学习然后删除
5. **暴露状态。** 每次操作后（逻辑）或每次变体切换后（UI），打印/渲染完整状态，让用户看到变化
6. **用完即删或吸收。** 原型回答了问题之后，要么删除，要么把验证过的决策吸收进真实代码——不要留在仓库腐烂

#### 原型结束后的产出

唯一值得保留的是**答案**。记录下来（commit message、ADR、Issue、或原型旁边的 `NOTES.md`），包括：
- 回答了什么问题
- 结论是什么
- 为什么不选其他方案

如果用户在线，一句话确认就够了；如果不在线，留个占位符等下次填充。

### B.4 架构健康自检

> 代码在不知不觉中变"泥球"。定期自检，在还便宜的时候修正。

**触发时机（满足任一即可）：**
- 完成一个 milestone 后
- 新增/修改了超过 3 个模块后
- 用户感觉"代码变乱了"时

**自检清单：**

- [ ] **深浅检查：** 最近的改动里，有没有模块的接口（参数数量、方法数量）
       接近或超过其内部实现复杂度？→ 那是浅模块，需要加深
- [ ] **删除测试：** 对最近改动的模块问——删掉它，复杂度是集中消失了，
       还是散到了 N 个调用方？
       - 消失 = earned its keep（模块有深度）
       - 散开 = pass-through（模块是多余的透传层）
- [ ] **耦合检查：** 有没有新增的 circular dependency 或跨层调用？
- [ ] **测试可达性：** 改动涉及的模块，有没有合适的测试 seam？
       如果某个模块"想测但测不了"，记录这个问题

**自检输出：** 列出 1-3 个最深的问题，说明影响和修复方向。
不要列出 10 个问题——那等于没有重点。
